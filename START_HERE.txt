DEWARP PIPELINE — v72
====================
(Scanned art-plate reconstruction.)
Deskews, crops and whitens scanned art-annual pages so each printed artwork is
axis-aligned (on its ORIGINAL ground -- whitened paper for paper pages, the kept BLACK ground for art-on-black pages), with all page text retained. Drop this
folder into a session to run.

>>> HANDOVER NAMING — DO THIS EVERY TIME <<<
  This pipeline is called "Dewarp Pipeline". Whenever you produce a new handover
  zip, name it  Dewarp_Pipeline_v<N>.zip  where <N> is the previous version + 1.
  This build is v71, so the next handover is v72, then v73, and so on — never reuse a
  number, never drop the name. After bumping it, update the "DEWARP PIPELINE — v<N>"
  line at the top of this file to match, so the version always travels with the code.
  This build is v72; the next handover is v73.

>>> READ FIRST — THIS IS AN IMAGE PIPELINE <<<
  The deliverable is RENDERED JPEG FILES, never text. When the user posts page
  scans to fix: stage them, run reconstruct.py, and present the resulting
  *_reconstructed.jpg from OUTPUT_DIR with the file-presentation tool. Do NOT
  describe, summarise, or "show" results as text or code output — actually write
  the image files and surface them. Verify each output visually before shipping.
  Do NOT render a CONTACT SHEET / thumbnail montage / overview grid on the user's
  behalf -- it is not part of the deliverable. Ship the per-page *_reconstructed.jpg
  (or a zip of them for a large batch) only. Make a contact sheet ONLY if the user
  explicitly asks for one.
  >>> NO SAMPLES (v27) <<< The deliverable is the FULL set of per-page
  *_reconstructed.jpg (or one zip of all of them) -- nothing else. Do NOT produce,
  copy out, or "present" a handful of SAMPLE / preview / representative pages, and
  do NOT ask the user whether they want samples. Run the whole batch, zip it, ship
  the zip. (Internal QC crops are fine to LOOK at while verifying, but they are not
  delivered and not surfaced to the user.) The user will say if changes are wanted;
  do not solicit per-page decisions.
  Environment note: numpy is 2.x — use np.ptp(arr), NOT arr.ptp() (method removed).
  Deps already expected: opencv-python (cv2), numpy, scipy.

  - NEW v54 — MAGENTA-GUIDED TEXT-BLOCK CORRECTION (new module magenta_text.py).
    Adds the ability to correct a TEXT column from a magenta guide, alongside the
    existing art-box magenta workflow. The art path (magenta_crop.magenta_dewarp) is
    UNCHANGED; text correction is layered on top, so art-only markups behave exactly
    as in v53 (the text pass is a no-op when no text region is present).
      HOW A TEXT GUIDE IS DRAWN. On the pixel-aligned clean copy, draw a magenta
      guide around the text column. It need NOT be a closed box:
        - left-justified text  -> usually NO right guide (open-right "C")
        - right-justified text -> may have NO left guide
        - any / all four sides may be present.
      The TOP stroke is NOT a crop edge: it traces how the top of the text is
      skewed/warped. The block is FLATTENED so the top guide (and the bottom guide,
      if drawn) becomes straight and level. The flatten is applied to the ALREADY-
      COMPOSED, page-whitened page, so the block keeps the PAGE's own background tone
      (no separate whitening pass; exposed edges replicate the page border) — no
      bright rectangle, no seam. A page_dewarp-style column remap keyed to the
      artist's guide instead of detected baselines.
      TEXT vs ART is decided by CONTENT (region_is_text: low colour + regular text
      lines in the CLEAN original inside the region), so a guide over prose is
      corrected and a guide over a plate is dewarped as art. Magenta pixels only
      ever supply geometry; none reach the output.
      REGION GROUPING (also fixes the recurring detached-edge case). magenta_text
      .magenta_regions() assembles magenta strokes into regions by SPAN-REACH: a
      horizontal and a vertical are two sides of the same box if the H's x-span
      reaches the V's column AND the V's y-span reaches the H's row. Only H-V links,
      so adjacent plates sharing a gutter stay separate, while an OPEN box, an offset
      corner, or a small corner gap still groups. This recovers boxes that the
      closed-loop magenta_boxes drops. (Very large corner gaps — hundreds of px —
      still won't bridge; close the corner when marking.)

    the FIRST deskew can under-rotate a plate whose printed edge is soft (a grey

    art tilted by ~1-3 deg -- the wedge IS the page gap beside that still-tilted

    art so it fills the rectangle), not paint-over. So: (1) deskew once (pass 1);

    crop boundary). A whole-edge fit is washed out to ~0 when the opposite side is

    the art must FILL its quad (>=0.86 -> skip L-shaped/garbage components); and the
    corner pull must be a real-but-plausible 1.2%-6% of the plate (skip clean plates
    and detection blow-ups like a 17% deviation). Verified: 032 top triangle closed
    (collage now a clean rectangle, caption/header kept), 069/036 corner triangles
    closed, art intact; clean pages (027 after its rotation fix, 020, 031, 060) are
    left untouched. Runs per plate (compose_multi) so each picture is corrected
    independently. ORDER end to end: deskew -> rotation re-dewarp -> close_triangles
    (keystone) -> compose -> edge-sliver fill (soft residue).

  - GAP-AWARE PLATE SPLIT (v13, deskew_crop.art_pictures via _merge_boxes_gapaware
    + _should_merge): the biggest cause of a remaining 'triangle' on a multi-plate
    page was NOT the dewarp -- it was plate DETECTION. Two separate paintings that
    sit close together (e.g. 052 fire-angel over warrior, 070 cat-portal over egg,
    and the 3-up pages 010/060) were bbox-MERGED into one box because their gap was
    just under the merge gutter. The single merged box then deskews to the DOMINANT
    painting and the second one is never isolated -> it keeps its scan tilt -> the
    'triangle' beside its still-tilted edge. Fix: before merging two near boxes,
    look at the strip BETWEEN them -- if it is page-coloured (art coverage < 0.22)
    they are SEPARATE plates and stay split; only merge when the gap actually holds
    art (one artwork split by a thin band). Once split, each plate runs through the
    normal deskew + rotation re-dewarp + close_triangles independently and comes out
    straight (verified: isolated 052 fire-angel deskews to 0.0 deg on its own). This
    is why most flagged pages read 'clean' per-plate yet showed triangles in the
    composed page: the tilted plate was never given its own deskew. Tuned so it only
    splits genuine multi-plate pages (010/052/060/070...): single paintings, whose
    internal light areas are not full page-gaps, are untouched.

  - PAINT-OVER REMOVED (v14, EDGE_SLIVER_FIX=False in reconstruct.py): the edge-
    sliver fill (deskew_crop.remove_edge_slivers) is no longer wired in. Rationale
    (Gique): a properly dewarped plate fills its rectangle, so there is NO sliver to
    remove -- a leftover sliver means the dewarp under-reached and must be fixed at
    the dewarp (rotation re-dewarp / close_triangles / plate split), NOT hidden with
    paper texture. Verified after removal: 027 and 069 left edges are clean, sharp,
    straight art-to-paper transitions with no grey band; nothing reappeared. The
    function is kept in deskew_crop.py for reference but is dead code now. If a grey
    scanner-transition band ever shows on an otherwise-straight edge, fix the CROP
    (tighten the inset so the crop edge lands inside solid art), do not re-enable a
    paint step.
  NOTE: there is NO white-text-on-black colour-bleed step in this pipeline. If a
  dark page with white text needs that, it must be added and tested on a real page
  first. (Do not import it from any other project's engine.)

WHAT IT DOES
------------
Each page is classified as one of:
  FULL_BLEED   one or more art rectangles on shaded paper (the common case)
  DARK_BG      art on a dark ground; panels composited onto a flat margin-tone canvas
  TEXT_LEFT    art with a text column; panels composited onto the flat-field-whitened
               page (real paper TEXTURE + caption text + page numbers kept), NOT a
               flat fill -- so margins/gaps show paper grain and any over-crop reveals
               the original art underneath instead of a blank patch
  TEXT         full text page (contents / colophon / essay): curl-dewarp +
               flat-field whiten, original pixels relocated, NO OCR
  SKIP         not reconstructable as panels

FULL_BLEED handling (the core path):
  1. Detect art as the largest NON-PAPER component, using an off-white
     paper profile (per-page value anchor + global saturation/value bounds; hue
     ignored). Isolates art from caption / header / paper, even on partial-page
     plates.
  2. Deskew: rotate by an angle estimated ONLY from reliable border edges (noisy
     dark-on-dark edges are discarded, so no over-rotation).
  3. Crop: bounding box of the deskewed art plus a small uniform inward inset, so
     the straight rectangle sits just inside the art (no paper sliver / skew
     wedge / tilted keyline).
  4. Compose: flat-field-whiten the page (removes vignette, keeps paper texture
     AND all printed text -- page numbers, plate numbers, captions, signatures),
     erase the original skewed art footprint, paste the clean deskewed art on top.
     The erased footprint is filled with _bg_texture (a generous sampled patch of
     paper, mirror-tiled) -- NOT a flat average colour -- so no flat fill survives.
  5. Patch-erase: wipe a thin frame just outside the pasted art to remove orphan
     fringe protruding past the straight edge (also filled with _bg_texture).
     Margin text survives via a connectivity rule (small non-paper islands not
     joined to the art mass kept). DARK_BG canvases use _bg_texture(paper=False),
     sampling the dark ground's grain instead of a flat colour.


MULTI-PICTURE PAGES (index / showcase pages with 2+ artworks)
-------------------------------------------------------------
Some FULL_BLEED-classified pages carry several separate artworks plus a caption
column. The pipeline detects and straightens EACH artwork independently:

  art_pictures(orig, bg) -> list of full-res [x0,y0,x1,y1] picture boxes (>=1).
    A "picture" is a sizeable component whose pixels are MOSTLY strong art
    (clearly past the paper envelope: sat > s_max+30 OR
    value < page_whitepoint-dv-30). That strong-coverage test (keep >= 0.45)
    separates a real artwork -- which scores high even with a light/pale
    background -- from soft paper vignette/shadow blobs and thin page-edge
    slivers, which score low and are rejected. Components of one split painting
    are bbox-merged so a single artwork is never returned as several pictures.

  compose_multi(orig, bg, flatfield_whiten, boxes):
    Whiten the page once (caption column / headers / plate+page numbers kept),
    then deskew + inward-crop each picture box independently and paste each back
    over its own footprint. Text between/around the pictures is never inside a
    footprint, so it survives untouched.

Routing (FULL_BLEED branch):
    pics = art_pictures(img, bg)
    len(pics) >= 2 -> compose_multi      len(pics) == 1 -> compose_fullbleed
  compose_fullbleed self-detects its own art region; the passed box is ignored,
  so single-artwork output is unaffected by the multi path.


PAGE-CURL CORRECTION (pre-pass, page_dewarp.py)
-----------------------------------------------
Deskew only ROTATES (handles per-picture SLOPE); it cannot remove binding-curl, the
smooth vertical bow that leaves picture top/bottom edges arced. So curl is removed
at the PAGE level, on the original scan, BEFORE any picture is detected or cropped
(wired in via PAGE_DEWARP=True, applied right after classify). Then the existing
per-picture deskew runs on an already-flat page. Clean split of labour:
  page field  -> the smooth curl (the bow)
  per-pic deskew -> each picture's individual tilt (the slope)
TEXT_LEFT / DARK_BG panels are NOT rotated by clean_crop (straight cut only), so
they get their own deslope: DESLOPE_TEXT=True wraps clean_crop with deslope_crop,
which measures each panel's _reliable_angle (same gate as the FULL_BLEED deskew),
rotates a padded region about the box centre when the tilt is real, then makes the
straight cut on the upright panel. So the curl/slope split now holds for EVERY
page type, and the page field is free to do curvature only.

Why it is reliable where the earlier per-edge "flatten" was not: it never traces
an edge column-by-column. Each picture border (from art_pictures boxes) is fit to
a low-order curve with a RESIDUAL GATE -- exactly how _reliable_angle trusts or
rejects a slope. A border that is not a clean line (a treeline, a sky fading into
paper) fails the gate and is discarded, not followed. The surviving clean borders
from several pictures, at different heights, over-constrain ONE low-order field
[1,x,y,x^2,(y^2),xy] (y^2 only with >=3 edges). Too smooth to bow a straight edge;
too over-constrained for a single bad border to move. Needs >=2 clean borders that
agree, so single-picture pages auto-no-op and fall through to compose_fullbleed
(which already handles a lone plate as one unit -- why single-image pages were
always clean). Pure vertical remap, so caption columns only shift a few px.
Knobs: page_dewarp.dewarp_page(min_field=4.0); _edge_samples(resid_thr=3.5).


KNOWN-BUG FIXES BAKED IN (do not regress)
-----------------------------------------
  - stem is assigned ONCE at the top of the page loop. Previously it was set only
    inside the TEXT / FULL_BLEED branches, so a TEXT_LEFT or DARK_BG page inherited
    the PREVIOUS page's stem and silently overwrote that page's output (no error).
    Any batch run with the old script may have lost a plate wherever a TEXT_LEFT
    page followed another page -- worth spot-checking old outputs.
  - PAGE_OVERRIDES can force a page type; e.g. a showcase page whose only frameless
    cutout fools the classifier into SKIP can be pinned to FULL_BLEED.


MAGENTA-OUTLINE OVERRIDE (magenta_crop.py)
------------------------------------------
Fix-a-bad-page workflow: when auto-detection mis-finds a border, take the rendered
page, draw the correct art borders on it as a CONTINUOUS outline in one specific
MAGENTA (the gate is HSV hue ~163, high sat -- a bright pink-magenta; keep it that
exact colour and keep each loop closed), and re-submit.

  >>> The magenta is drawn on a COPY. Draw it on a copy of the EXACT original scan
      so the markup and the original are PIXEL-ALIGNED, and KEEP the original --
      the markup gives the boxes, the original gives the pixels. <<<

The pipeline detects the magenta loops (has_magenta + magenta_boxes), takes each
closed loop as one art rectangle (the loop's INNER edge), then reconstructs from
the CLEAN ORIGINAL: magenta_dewarp() runs the full per-plate deskew + inward-crop
on the original within those boxes and composites onto the whitened original page,
KEEPING caption text + page numbers outside the loops. The markup's (re-compressed)
pixels are never used; the magenta line never has to be scrubbed from the output
because the output is built from the clean original.
  - WHY: a magenta page is a markup copy, normally lower-quality and with ink over
    the art. Dewarping it would ship a degraded, un-straightened plate. Boxes-only
    + reconstruct-from-original gives a full-quality, properly dewarped result.
  - NO ORIGINAL? STOP and ask for it. The batch run will NOT reconstruct a markup
    that has no staged original — it saves the boxes to <stem>.magboxes.json and
    asks for the original. Never hand-wave this by dewarping the markup.
  - Pairing in a batch: stage the markup AND its clean original together. They are
    paired by normalised stem (download-dedup suffixes __2_/(1) and _orig/_clean/
    _mag tags stripped) + identical dimensions + the candidate having no magenta
    loops. The original is consumed (not processed as a standalone page); the output
    is <normalised-stem>_reconstructed.jpg.
  - Works for single-image, multi-image and text-column pages (loops may touch /
    share edges, as in a 2x2 grid). Small breaks in the line are bridged
    (directional close); cleaner closed lines = cleaner boxes. Because the original
    is re-deskewed/inward-cropped per box, a slightly loose loop is fine — the crop
    snaps to the real printed art edge — but drawing close to the art is still best.
  - Standalone (proper, ships):
      python3 magenta_crop.py --original CLEAN_ORIGINAL.jpg page_with_magenta.jpg
      # -> CLEAN_ORIGINAL_reconstructed.jpg  (boxes from markup, pixels from original)
  - Standalone (PREVIEW ONLY, do not ship — literal crop off the markup, no dewarp):
      python3 magenta_crop.py page_with_magenta.jpg   # -> *_magcrop.jpg


RUN
---
  1. Stage the WHOLE batch of input JPEGs into INPUT_DIR (default
     /home/claude/work/input) -- all the pages you want one shared profile over
     (e.g. the ~80 pages of a handover). Outputs (*_reconstructed.jpg, *_text.jpg)
     go to OUTPUT_DIR (default /home/claude/work/output). Edit both paths at the
     top of reconstruct.py if needed. Flags PAGE_DEWARP / DESLOPE_TEXT /
     MAGENTA_OVERRIDE / RESUME are at the top of the same file (all default True).
  2. python3 reconstruct.py
     The profile is rebuilt automatically at the start of EVERY run, as an envelope
     over whatever is staged in INPUT_DIR -- no separate build step, no cache. So
     the threshold is always consistent across the staged set, and never inherited
     from a previous run. (build_profile.py is optional, only if you want to inspect
     the numbers for a folder first.)
  3. COMPUTE WINDOWS: each plate is ~13-15 s (flat-field inpaint + 2x warpAffine +
     footprint mask), so ~80 pages can exceed one window. DO NOT clear INPUT_DIR to
     chunk -- that would shrink the profile to the chunk. Instead keep the whole
     batch staged and just RUN AGAIN: RESUME skips pages already rendered, so each
     run rebuilds the profile over the full set (identical numbers) and renders only
     what is left. Repeat until every page is done.
  4. Deliverable: rendered *_reconstructed.jpg in OUTPUT_DIR. Verify visually before
     shipping.


PAPER PROFILE (off-white range)
-------------------------------
build_paper_profile(paths) takes an ENVELOPE across pages of each page's paper
(margin-ring sample; outer rim dropped), covering grey/beige/blue/yellow stock,
plus slack pushed close to the over-identification edge:
    s_max = p95(per-page paper saturation-p98) + 44
    dv    = p95(per-page paper value-drop)      + 30
Hue is ignored (capping saturation already accepts any tint). Per-page value
anchoring (white point from each page's own margins) handles brighter/darker
pages individually.

paper_profile.json is rebuilt over the staged set on every run (see v3 note above),
so it is always valid for whatever is currently in INPUT_DIR -- there is no stale
cache to worry about. Stage the WHOLE batch you want one threshold over so the
envelope is built across all of it at once. If a page has a very pale-edged
painting that starts clipping (art region eaten into), lower the slack until that
plate's art stops clipping.


TUNING KNOBS
------------
  deskew_deep_crop(margin_frac=0.015)   inward bite depth; raise to crop more
                                        aggressively into the art, lower to keep
                                        more (incl. printed keylines/borders).
  art_pictures(min_area=0.035, min_dim=0.11, min_strong=0.45)
                                        min_strong: lower to admit very
                                        pale-backgrounded artworks; raise if a
                                        tinted-paper blob is mistaken for art.
  PAGE_OVERRIDES (top of reconstruct.py)
                                        force a page type for a filename the
                                        classifier gets wrong.

Sanity-check, before shipping a new batch, that art_pictures returns the
expected picture count on any index/showcase pages (a single painting should
return 1; a true multi-artwork page should return its real count).
